# 错误处理策略改进方案  
版本：v1.0  日期：2025-07-13  

## 1. 设计原则  
1. 可观测性：任何错误都必须被**捕获-记录-告警**，并带完整上下文（trace_id）。  
2. 可隔离性：单点故障不能级联；外部依赖异常时快速熔断。  
3. 可恢复性：自动重试 + 幂等保证，失败后可重放。  
4. 可降级性：在部分依赖失效时维持核心交易功能。  

---

## 2. 断路器（Circuit Breaker）  
| 场景 | 依赖 | 方案 | 关键参数 |
|------|------|------|----------|
| ① 下单 / 查询行情 | OKX REST & WS | `pybreaker` 三态断路器（Closed → Open → Half-Open） | failure_threshold=5, recovery_timeout=30 s |
| ② 数据库写入 | TimescaleDB | 自研轻量断路逻辑：连续 3 次 `OperationalError` 时熔断 | 熔断 10 s 后探针再试 |
| ③ Redis / Celery Broker | Redis | Celery 内置 `broker_pool_limit=0` + 监控告警， 超时后切本地队列 | 超时阈值 2 s |

示例（OKX API 包装层）：  
```python
import pybreaker, requests
breaker = pybreaker.CircuitBreaker(
    fail_max=5, reset_timeout=30,
    listeners=[PrometheusBreakerListener()]
)

@breaker
def place_order(req: OrderRequest):
    return requests.post(OKX_URL, json=req.dict(), timeout=2)
```

## 3. 重试逻辑  
1. **库**：`tenacity`（同步） + Celery `autoretry_for`（异步任务）。  
2. **梯度退避**：指数退避 `wait=wait_exponential(multiplier=0.5, max=30)`.  
3. **幂等保障**  
   - REST 下单使用 `client_order_id` (UUID4)；  
   - 数据库写入使用 UPSERT + 唯一键 (order_id, account_id)。  

Celery 任务示例：  
```python
from celery import shared_task
from tenacity import retry, stop_after_attempt, wait_exponential

@shared_task(bind=True, autoretry_for=(Exception,), retry_backoff=True,
             retry_kwargs={'max_retries': 5})
def sync_account_status(self, account_id):
    _sync_impl(account_id)

@retry(stop=stop_after_attempt(5), wait=wait_exponential())
def _sync_impl(account_id):
    ...
```

## 4. 优雅降级策略  
| 组件 | 异常场景 | 降级行为 |
|------|----------|----------|
| Web API (FastAPI) | Redis 不可用 | 切换本地内存缓存，返回 `warning: degraded_cache` |
| Vue 前端 | `/api/v1/trading/positions` 超时 | Toast 提示 + 显示缓存快照，30 s 后自动重拉 |
| 策略执行 | 交易所接口重试失败 | 标记策略为 `paused`，写 ALERT，等待人工恢复 |
| 备份任务 | OSS 上传失败 | 本地保留备份文件并重试 1h；超 24h 触发告警 |

## 5. 监控与预警  
### 5.1 指标  
- FastAPI：`prometheus-fastapi-instrumentator`  
- Celery：`celery-exporter`（任务失败率、运行时）  
- 系统：CPU/Mem/Disk via node-exporter  

### 5.2 日志  
- 格式：JSON（生产）/Text（开发）  
- handler：`TimedRotatingFileHandler` 每日切分，保留 14 天  
- trace_id：使用 `structlog` + `uuid4` 注入到 Request & Celery task context  

### 5.3 告警  
- Alertmanager 路由到 企业微信 Webhook，级别映射：  
  - CRITICAL：@all + 电话  
  - ERROR：值班人员  
  - WARNING：IM 通知  
- FastAPI 全局异常中间件 & Celery `task_failure` 信号 → Push to Alertmanager API  

---

## 6. 统一异常处理层  
```
┌──────────────┐
│ Presentation │  Vue ↔ FastAPI
├──────────────┤
│  Service      FastAPI Router (Try/Catch → HTTPException, trace_id)
├──────────────┤
│  Domain       Strategy/Exchange (Custom Exceptions)
├──────────────┤
│  Infra        OKX SDK / SQL / Redis
└──────────────┘
```
- 自定义 `AppError(code, msg, detail)` 继承 `Exception`；  
- FastAPI `exception_handler(AppError)` → 4xx/5xx JSON。

## 8. 风险与缓解  
1. **断路器误触**：初期阈值偏低→灰度环境 A/B 调整。  
2. **重试风暴**：统一使用 `rate_limit` & `jitter`；全局并发控制。  
3. **告警噪音**：Alertmanager 抑制 + 聚合策略，避免重复。  

---

## 9. 结语  
实施以上改进后，可确保：  
- 外部依赖故障平均恢复时间（MTTR） < 5 min  
- 关键接口 99.9% 可用性  
- 错误可全部追溯，告警实时可见  
- 系统在降级状态下仍能执行核心交易与风控逻辑  
这将为后续多交易所扩展、资金规模提升提供稳健保障。
