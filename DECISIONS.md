## ADR-001: 技术栈选型

### 状态

- [x] 已接受
- [ ] 已驳回
- [ ] 已废弃

### 上下文

项目需要选择一套完整的技术栈来构建数字货币量化交易 App。需要考虑：
- 移动端跨平台能力
- 后端性能和可扩展性
- 实时数据处理能力
- 团队技术储备

### 决策

采用以下技术栈：

| 层级 | 技术选型 | 备选 |
|------|----------|------|
| 移动端 | **Flutter (Dart)** | React Native |
| 状态管理 | **Riverpod** | BLoC |
| 后端框架 | **Python FastAPI** | Go Gin / Node NestJS |
| 主数据库 | **PostgreSQL** | MySQL |
| 时序数据 | **TimescaleDB** | InfluxDB |
| 缓存 | **Redis** | - |
| 消息队列 | **Redis Streams** | RabbitMQ |
| K线数据源 | **Binance WebSocket API** | CoinGecko / CCXT |
| 部署 | **Docker + 云服务器** | Serverless |

### 备选方案

1. **React Native**：更广泛的 JS 生态，但性能略逊于 Flutter
2. **Go Gin**：性能更高，但学习曲线陡峭
3. **InfluxDB**：专用时序数据库，但 PostgreSQL 生态更成熟

### 后果

**正面影响：**
- Flutter 跨平台能力强，一次开发 iOS/Android 双端
- FastAPI 现代、高性能、自动生成 OpenAPI 文档
- TimescaleDB 在 PostgreSQL 基础上支持时序数据，运维简单

**负面影响：**
- Flutter 团队需要学习 Dart 语言
- 量化计算 Python 性能不如 C++/Go，但足够

### 日期

2026-04-21

### 决策者

@architect, @pm

## ADR-002: 状态管理方案

### 状态

- [x] 已接受
- [ ] 已驳回
- [ ] 已废弃

### 上下文

Flutter 应用需要管理复杂的状态，包括：
- 用户认证状态
- 账户资产数据
- 策略配置
- 实时行情数据
- UI 交互状态

### 决策

选择 **Riverpod** 作为 Flutter 状态管理方案。

### 备选方案

1. **BLoC**：更严格的分层，但代码量较大
2. **Provider**：简单，但大型应用下难以维护
3. **setState**：仅适用于简单场景

### 后果

**正面影响：**
- Riverpod 类型安全、易测试
- 编译时检查减少运行时错误
- 易于处理异步数据和依赖注入

**负面影响：**
- 学习曲线较陡
- 需要理解 Provider 的各种类型

### 日期

2026-04-21

### 决策者

@architect, @frontend

## ADR-003: 量化引擎部署位置

### 状态

- [x] 已接受
- [ ] 已驳回
- [ ] 已废弃

### 上下文

量化策略引擎需要考虑的计算资源：
- K线数据处理
- 策略信号计算
- 回测引擎（CPU 密集）

### 决策

量化引擎采用 **云端部署** 方案。

### 备选方案

1. **端侧运行**：保护策略隐私，但设备性能受限
2. **混合方案**：简单策略端侧，复杂策略云端

### 后果

**正面影响：**
- 不受移动设备性能限制
- 便于策略更新和维护
- 统一管理便于风控

**负面影响：**
- 策略逻辑需要上传，存在泄露风险
- 依赖网络连接

## ADR-004: 数据库选型

### 状态

- [x] 已接受
- [ ] 已驳回
- [ ] 已废弃

### 上下文

系统需要存储：
- 结构化业务数据（用户、策略、订单）- **关系型**
- 时序行情数据（K线、Tick）- **时序型**
- 缓存数据（会话、实时行情）- **缓存型**

### 决策

采用 **PostgreSQL + TimescaleDB** 作为主数据库：

| 数据类型 | 存储方案 |
|----------|----------|
| 业务数据 | PostgreSQL |
| K线历史 | TimescaleDB (Hypertable) |
| 缓存 | Redis |

### 备选方案

1. **MySQL + InfluxDB**：更常见，但两个系统增加运维复杂度
2. **MongoDB**：灵活但不适合强一致性业务
3. **纯 TimescaleDB**：时序优先但业务查询不够灵活

### 后果

**正面影响：**
- TimescaleDB 基于 PostgreSQL，运维统一
- 支持全文搜索、JSON 等高级特性
- 水平扩展能力强

**负面影响：**
- TimescaleDB 需要额外学习
- 需要定期维护 Hypertable 分区

### 日期

2026-04-21

### 决策者

@architect, @backend

## ADR-005: 消息队列选型

### 状态

- [x] 已接受
- [ ] 已驳回
- [ ] 已废弃

### 上下文

系统需要处理：
- 实时行情推送
- 订单状态更新
- 信号通知
- 风控事件

### 决策

选择 **Redis Streams** 作为消息队列。

### 备选方案

1. **RabbitMQ**：功能更丰富，但运维复杂
2. **Kafka**：高吞吐，但过于重量级
3. **Celery**：Python 生态友好，但非专业 MQ

### 后果

**正面影响：**
- Redis 已在使用，零额外依赖
- 支持消费者组、消息持久化
- 延迟极低，适合实时场景

**负面影响：**
- 不适合超大规模消息场景
- 功能不如 Kafka/RabbitMQ 丰富

## ADR-006: 安全审计修复（P0/P1问题清零）

### 状态

- [x] 已接受
- [ ] 已驳回
- [ ] 已废弃

### 上下文

2026-04-21 全量代码审计发现大量安全问题，包括：
- 硬编码默认密钥/凭证
- API Key 未加密存储
- 多个端点缺失认证
- Refresh Token 未验证类型
- 金融数值缺少范围校验
- 回测使用随机数伪造指标
- 代码重复、N+1查询、性能隐患等

### 决策

按四级优先级全面修复：

| 等级 | 修复项数 | 关键变更 |
|------|---------|---------|
| P0 | 6 | 移除硬编码密钥默认值、添加AES-256加密、端点加认证、Token类型校验、数值校验 |
| P1 | 18 | 修复模型字段不一致、去重代码、httpx单例复用、Redis锁保护、结构化日志等 |
| P2 | 5 | Redis行情缓存、提取常量、依赖注入规范化 |
| P3 | 1 | 更新DECISIONS.md |

### 后果

**正面影响:**
- 安全漏洞全部清零，可安全上线
- 代码重复消除，维护性大幅提升
- 性能隐患修复（httpx复用、N+1优化）

**负面影响:**
- 需要数据库迁移（新增 balance/frozen_balance/pnl 字段）
- 需要安装 cryptography 依赖
- 现有 API Key 需要重新加密存储

### 日期

2026-04-21

### 决策者

@architect, @backend
