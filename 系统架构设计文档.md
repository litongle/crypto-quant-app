## 1. 架构概览

### 1.1 设计原则

```
┌─────────────────────────────────────────────────────────────┐
│                    架构设计四大原则                            │
├─────────────────────────────────────────────────────────────┤
│  ① 策略安全第一  ─  密钥不出托管环境，风控先于利润            │
│  ② 领域驱动设计  ─  交易逻辑与业务边界清晰隔离                │
│  ③ 事件驱动架构  ─  响应市场变化，支持异步解耦                │
│  ④ 可观测优先    ─  交易系统必须有完整的日志、指标、追踪       │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 上下文依赖关系

| 上下文 | 依赖方 | 依赖类型 | 集成方式 |
|--------|--------|----------|----------|
| 用户上下文 | 所有上下文 | 基础认证 | REST + JWT |
| 账户上下文 | 交易上下文、风控上下文 | 余额校验 | 事件驱动 |
| 行情上下文 | 策略上下文、交易上下文、回测上下文 | 市场数据 | WebSocket + Kafka |
| 策略上下文 | 账户上下文、行情上下文 | 信号输出 | 事件驱动 |
| 交易上下文 | 账户上下文、交易所适配 | 订单执行 | API 调用 |
| 风控上下文 | 所有上下文 | 风控校验 | Sidecar + 事件 |
| 回测上下文 | 行情上下文、策略上下文 | 历史数据 | 批处理任务 |

### 3.3 风控多层架构

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│                           多层风控架构 (Multi-Layer Risk Control)                │
├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                 │
│                        信号生成 ──► 层层过滤 ──► 订单执行                         │
│                                                                                 │
│  ┌─────────────────────────────────────────────────────────────────────────┐   │
│  │  Layer 1: 用户层风控 (User-Level Risk)                                  │   │
│  │  ┌─────────────────────────────────────────────────────────────────┐   │   │
│  │  │  • 单币种最大持仓        (如：BTC ≤ $10,000)                    │   │   │
│  │  │  • 单笔最大下单金额      (如：$500/笔)                          │   │   │
│  │  │  • 日最大交易次数        (如：20次/天)                          │   │   │
│  │  │  • 日最大亏损上限        (如：-5% 强平所有仓位)                  │   │   │
│  │  │  • 总仓位上限           (如：占用保证金 ≤ $50,000)              │   │   │
│  │  └─────────────────────────────────────────────────────────────────┘   │   │
│  └─────────────────────────────────────────────────────────────────────────┘   │
│                                      │                                           │
│                                      ▼                                           │
│  ┌─────────────────────────────────────────────────────────────────────────┐   │
│  │  Layer 2: 策略层风控 (Strategy-Level Risk)                              │   │
│  │  ┌─────────────────────────────────────────────────────────────────┐   │   │
│  │  │  • 策略止损 (固定止损/追踪止损)                                  │   │   │
│  │  │  • 策略止盈 (固定/分批)                                         │   │   │
│  │  │  • 最大连续亏损次数 (如：5连亏 → 暂停策略)                       │   │   │

│  │  └─────────────────────────────────────────────────────────────────┘   │   │
│  └─────────────────────────────────────────────────────────────────────────┘   │
│                                      │                                           │
│                                      ▼                                           │
│  ┌─────────────────────────────────────────────────────────────────────────┐   │
│  │  Layer 3: 系统层风控 (System-Level Risk)                                │   │
│  │  ┌─────────────────────────────────────────────────────────────────┐   │   │
```

## 8. 架构决策记录 (ADR)

### ADR-001: 整体架构模式选择

**状态**：已接受

**上下文**：
10 人全栈 AI 团队，全球市场，需要支持多交易所、多策略、实时交易。

**决策**：
采用**微服务架构 + 事件驱动**，按业务领域拆分服务，通过消息队列解耦。

**选项分析**：

| 选项 | 优势 | 劣势 | 适用场景 |
|------|------|------|----------|
| 单体应用 | 简单、部署快 | 扩展难、技术栈受限 | MVP < 5 人 |
| 微服务 | 独立扩缩容、团队自治 | 复杂度高、运维成本高 | 中大型、有 DevOps 能力 |
| 模块化单体 | 平衡点、可演进 | 边界划分需经验 | 初期团队、演进路径清晰 |
| 事件驱动 | 松耦合、实时性好 | 调试难、事务复杂 | 量化交易、高并发 |

**后果**：
- **正面**：策略引擎独立演进、交易执行高可用、风控可插拔
- **负面**：需要服务治理（Istio）、消息队列运维、分布式追踪
- **需要新增**：SRE 角色或 DevOps 自动化

---

### ADR-002: 策略计算位置

**状态**：已接受

**上下文**：
用户配置策略后，需要 7×24 自动执行。包含网格、马丁格尔等低频策略，也包含均线、RSI 等中频策略。

**决策**：
**云端执行为主 + 端侧预览为辅**

- 回测引擎提供模拟计算能力（可在云端或浏览器 WebAssembly）
- 实时策略在云端执行，确保稳定性
- 移动端仅做状态展示和手动干预

**后果**：
- **正面**：不依赖用户手机、7×24 运行、可做复杂优化
- **负面**：API Key 必须托管在云端（需要信任）
- **缓解措施**：Key 加密存储、严格风控、审计日志

### ADR-003: 实时行情数据方案

**状态**：已接受

**上下文**：
量化交易需要实时 K 线、订单簿、成交数据。交易所：Binance、OKX、Huobi。

**决策**：
**自建行情聚合服务 + WebSocket 订阅**

- 每个交易所部署一个 Adapter 服务
- Adapter 通过 WebSocket 订阅交易所数据
- 数据通过 NATS 分发给其他服务
- TimescaleDB 存储 K 线历史

**选项分析**：

| 选项 | 延迟 | 成本 | 可靠性 |
|------|------|------|--------|
| 交易所 WebSocket 直连 | < 100ms | 免费 | 依赖单交易所 |
| 第三方数据商 (Kaiko/CoinMetrics) | < 500ms | $500+/月 | 高 |
| 自建聚合服务 | < 200ms | 服务器成本 | 可多源备份 |

**后果**：
- **正面**：成本可控、可多交易所聚合、可定制
- **负面**：需要维护多个交易所的适配代码
- **备选方案**：当自建服务不可用时，降级使用交易所 REST API 轮询

---

### ADR-004: 数据库选型

**状态**：已接受

**上下文**：
系统需要存储：用户数据（关系型）、K 线历史（时序）、实时状态（缓存）。

**决策**：
**PostgreSQL + TimescaleDB + Redis 三层架构**

- PostgreSQL：用户、账户、策略配置、订单等关系数据
- TimescaleDB：K 线、行情等时序数据（基于 PG，查询统一）
- Redis：实时持仓、信号缓存、会话、限流

**后果**：
- **正面**：统一技术栈、时序查询与关系查询可 Join、运维简单
- **负面**：时序场景下 PG 不如专业时序库（如 InfluxDB）
- **备选**：若 TimescaleDB 性能不足，可引入 InfluxDB 处理高频 K 线

---

### ADR-005: 风控执行模式

**状态**：已接受

**上下文**：
实盘交易涉及真实资金，必须有严格的风控机制。

**决策**：
**Sidecar 风控 + 策略级风控 + 全局风控三层**

- Sidecar：订单发出前拦截（同步）
- 策略级：止损止盈监控（异步事件驱动）
- 全局熔断：异常情况下暂停所有策略

**后果**：
- **正面**：多层次保护、风险可控、可追溯
- **负面**：增加响应延迟（约 5-10ms）
- **关键**：风控规则优先于策略执行
