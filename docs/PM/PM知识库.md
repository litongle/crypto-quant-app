## 一、敏捷方法论

### 1.1 Scrum 核心框架

```
┌─────────────────────────────────────────────────────────────┐
│                        Scrum 框架                            │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│   产品待办列表 (Product Backlog)                              │
│         ↓                                                    │
│   ┌─────────────────┐                                        │
│   │   Sprint 计划   │  ← Sprint Planning                     │
│   └────────┬────────┘                                        │
│            ↓                                                 │
│   ┌─────────────────┐                                        │
│   │   Sprint 待办   │  ← Sprint Backlog                     │
│   └────────┬────────┘                                        │
│            ↓                                                 │
│   ┌─────────────────┐                                        │
│   │   执行 Sprint   │  ← Daily Standup                       │
│   └────────┬────────┘                                        │
│            ↓                                                 │
│   ┌─────────────────┐                                        │
│   │   Sprint 评审   │  ← Sprint Review                      │
│   └────────┬────────┘                                        │
│            ↓                                                 │
│   ┌─────────────────┐                                        │
│   │   Sprint 回顾   │  ← Sprint Retrospective               │
│   └─────────────────┘                                        │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 每日站会最佳实践

**三问法则:**
1. 昨天完成了什么？
2. 今天计划做什么？
3. 有什么阻碍？

**站会规则:**
- 15 分钟以内
- 站立形式
- 每个人都要发言
- PM 不做汇报，而是引导
- 阻塞问题线下解决

### 1.3 迭代计划会议

**输入:**
- 产品待办列表
- 团队速率历史数据
- 团队可用时间

**输出:**
- Sprint 目标
- Sprint 待办列表
- 团队承诺

## 二、风险管理

### 2.1 风险识别技术

| 技术 | 适用场景 |
|------|----------|
| 头脑风暴 | 项目初期 |
| SWOT 分析 | 战略层面 |
| 检查表法 | 经验丰富的团队 |
| Delphi 法 | 需要专家意见 |
| 根本原因分析 | 问题复盘 |

### 2.2 风险评估矩阵

```
            影响
          低    中    高
概率  ┌─────────────────┐
 高   │ 🟡中 │ 🟠中高│ 🔴高 │
 中   │ 🟢低 │ 🟡中 │ 🟠中高│
 低   │ 🟢低 │ 🟢低 │ 🟡中 │
      └─────────────────┘
```

### 2.3 风险响应策略

| 风险类型 | 应对策略 |
|----------|----------|
| 高影响/高概率 | 规避或转移 |
| 高影响/低概率 | 应急计划 |
| 低影响/高概率 | 减轻 |
| 低影响/低概率 | 接受 |

## 三、需求管理

### 3.1 需求优先级框架

**MoSCoW 方法:**
- **M**ust Have (必须有): 核心功能，无则项目失败
- **S**hould Have (应该有): 重要但不致命
- **C**ould Have (可以有): 锦上添花
- **W**on't Have (不会有): 本次不考虑

**Kano 模型:**
```
满意度
  ↑
  │      兴奋型需求
  │          /
  │         /
  │        /   期望型需求
  │       /   /
  │      /   /
  │     /   /   基础型需求
  │    /   /   /
  │   /   /   /
  │  /   /   /
  │ /   /   /
  └─────────────────────→ 满足度
```

### 3.2 用户故事格式

```
作为 [用户角色]
我想要 [功能]
以便于 [价值/收益]
```

**验收标准:**
```
假设 [前提条件]
当 [触发事件]
那么 [期望结果]
```

## 四、团队协作

### 4.1 沟通矩阵

| 场景 | 渠道 | 频率 | 参与者 |
|------|------|------|--------|
| 日常同步 | 即时消息 | 随时 | 相关成员 |
| 进度跟踪 | 看板 | 每天 | 全团队 |
| 决策讨论 | 会议 | 按需 | 相关方 |
| 文档共享 | 知识库 | 按需 | 全团队 |

### 4.2 决策框架

**RAPID 模型:**
- **R**ecommend: 提出建议
- **A**gree: 同意签字
- **P**erform: 执行
- **I**nput: 提供意见
- **D**ecide: 做决定

### 4.3 冲突解决

1. **回避**: 暂时搁置，小问题
2. **妥协**: 各让一步
3. **调停**: 第三方调解
4. **强迫**: 紧急情况使用权力
5. **合作**: 最佳方案，共同解决

## 五、质量管理

### 5.1 质量门设置

```
需求 → 设计 → 开发 → 测试 → 部署
  ↓      ↓      ↓      ↓      ↓
 QG-0   QG-1   QG-2   QG-3   QG-4
```

每个质量门必须满足:
- 进入标准 (Entry Criteria)
- 验证清单 (Checklist)
- 退出标准 (Exit Criteria)

### 5.2 缺陷管理

**缺陷生命周期:**
```
新建 → 确认 → 指派 → 修复 → 验证 → 关闭
  ↓       ↓
 拒绝    延期
```

**严重程度:**
| 级别 | 定义 | 响应时间 |
|------|------|----------|
| S0 | 系统崩溃 | 立即 |
| S1 | 严重功能失效 | 4h |
| S2 | 一般功能缺陷 | 24h |
| S3 | 轻微问题 | 72h |

## 六、进度管理

### 6.1 估算技术

**敏捷估算:**
1. **规划扑克**: 斐波那契数列 (1, 2, 3, 5, 8, 13, 21...)
2. **三点估算**: (乐观 + 4×最可能 + 悲观) / 6
3. **相对估算**: 与已知任务比较

**速度计算:**
```
速度 = 本迭代完成的故事点数
平均速度 = 历史迭代速度的均值
预测 = 平均速度 × 迭代数
```

## 七、变更管理

### 7.1 变更控制流程

```
变更请求
    ↓
变更评估 (影响/成本/风险)
    ↓
CCB 审批 (变更控制委员会)
    ↓
┌──────────┐
│ 通过 → 实施 → 验证
│ 拒绝 → 记录 → 沟通
└──────────┘
```

### 7.2 范围蔓延预防

- 所有变更必须走正式流程
- 记录变更请求和理由
- 评估对进度/成本/质量的影响
- 获得干系人批准

## 八、干系人管理

### 8.1 干系人分析

| 干系人 | 利益 | 影响 | 策略 |
|--------|------|------|------|
| 项目发起人 | 业务成功 | 高 | 定期汇报 |
| 开发团队 | 工作环境 | 高 | 授权信任 |
| 用户 | 易用产品 | 中 | 持续反馈 |
| 管理层 | 成本控制 | 中 | 关键节点汇报 |

### 8.2 沟通计划

**5W1H 法:**
- **W**ho: 谁需要信息？
- **W**hat: 需要什么信息？
- **W**hen: 什么时候需要？
- **W**here: 通过什么渠道？
- **W**hy: 为什么需要？
- **H**ow: 怎么传递？

## 九、度量指标

### 9.1 敏捷度量

| 指标 | 计算 | 用途 |
|------|------|------|
| 速度 | 完成的 Story Points 总和 | 预测未来迭代 |
| 燃尽率 | 剩余工作量/时间 | 进度跟踪 |
| 缺陷逃逸率 | 生产缺陷数/总缺陷数 | 质量评估 |
| 需求变更率 | 变更次数/需求总数 | 范围控制 |

### 9.2 项目健康度

```
健康度 = f(进度, 质量, 风险, 团队, 沟通)

🟢 绿色: 所有指标正常
🟡 黄色: 1-2 个指标偏差
🔴 红色: 3+ 个指标严重偏差
```

## 十、持续改进

### 10.1 回顾会议

**Start / Stop / Continue:**
- **Start**: 开始做什么
- **Stop**: 停止做什么
- **Continue**: 继续做什么

### 10.2 改进闭环

```
识别问题 → 分析根因 → 制定措施 → 实施 → 验证 → 固化
```

### 10.3 经验教训

每个迭代结束后记录:
- 做得好的是什么
- 可以改进的是什么
- 学到的经验是什么
- 下次迭代要尝试什么
