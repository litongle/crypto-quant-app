## 二、风险详情

### R-01: 交易所 API 不稳定

**基本信息**

| 字段 | 内容 |
|------|------|
| 风险 ID | R-01 |
| 风险类别 | 技术风险 |
| 发现日期 | 2026-04-21 |
| 状态 | 🔴 活跃 |
| 负责人 | @backend |

**风险描述**

交易所（Binance/OKX）的 WebSocket API 可能出现连接不稳定、限流、断线等问题，导致行情数据中断，影响策略执行。

**影响分析**

| 维度 | 评估 |
|------|------|
| 影响程度 | 🔴 高 - 直接影响策略执行和用户体验 |
| 发生概率 | 🟡 中 - 历史上偶有发生 |
| 风险值 | 🔴 12 (高) |

**缓解策略**

```markdown
## 缓解措施

### 1. 主动防护
- [x] 实现 WebSocket 自动重连机制（指数退避）
- [x] 实施 API 限流保护
- [ ] 考虑 HTTP REST API 作为备选数据源

### 2. 数据冗余
- [ ] 接入多个交易所作为备选
- [ ] 考虑第三方行情数据服务商（如 CoinGecko）

### 3. 降级策略
- [ ] 当检测到数据异常时，自动切换到降级模式
- [ ] 向用户显示"数据延迟"提示
```

**监控指标**

- WebSocket 连接成功率 < 95% 时告警
- 数据延迟超过 5 秒时告警
- API 调用返回错误码超过 10 次/分钟时告警

**触发条件**

当以下任一条件满足时，风险升级：
- 连续 3 次重连失败
- 行情数据延迟超过 30 秒
- API 返回 429（限流）错误

**影响分析**

|------|------|
| 影响程度 | 🔴 高 - 可能导致用户资金损失 |
| 发生概率 | 🟡 中 - 取决于代码质量 |
| 风险值 | 🔴 12 (高) |

**缓解策略**

```markdown
## 缓解措施

### 1. 开发阶段
- [ ] 所有订单必须先在沙盒环境验证
- [ ] 实盘对接前必须通过 Paper Trading（模拟盘）验证
- [ ] 建立完整的单元测试和集成测试

### 2. 风控硬限制
- [ ] 单笔最大下单金额限制
- [ ] 单日最大亏损限制
- [ ] 持仓上限限制
- [ ] 异常交易自动拦截

- [ ] 大额订单（超过阈值）需二次确认
- [ ] 连续亏损时自动暂停策略
- [ ] 紧急情况下一键清仓

### 4. 用户教育
- [ ] 明确告知用户量化交易风险
- [ ] 提供模拟盘让用户先熟悉
```

**监控指标**

- 单笔订单金额超过 $100 时告警
- 单日亏损超过 5% 时暂停策略
- 连续 3 次订单失败时暂停策略

### R-03: 回测与实盘差距

**基本信息**

| 字段 | 内容 |
|------|------|
| 风险 ID | R-03 |
| 风险类别 | 业务风险 |
| 发现日期 | 2026-04-21 |
| 状态 | 🟠 已缓解（规划中） |
| 负责人 | @quant |

**风险描述**

量化策略在回测中的表现往往优于实盘，主要原因包括：滑点、执行延迟、流动性不足、市场冲击等。

**影响分析**

| 维度 | 评估 |
|------|------|
| 影响程度 | 🟡 中 - 影响策略可信度 |
| 发生概率 | 🔴 高 - 几乎必然发生 |
| 风险值 | 🟠 9 (中高) |

**缓解策略**

```markdown
## 缓解措施

### 1. 回测优化
- [x] 在回测中模拟滑点成本（建议 0.1%-0.2%）
- [x] 考虑手续费和交易所 Maker/Taker 费率
- [ ] 使用分价级别数据提高精度

### 2. 实盘验证
- [ ] 引入 Paper Trading（模拟盘）阶段
- [ ] 建议用户先用小额资金在实盘验证
- [ ] 对比 Paper Trading 与回测结果的差异

### 3. 透明告知
- [ ] 在回测报告中明确标注"回测结果不代表实盘表现"
- [ ] 提供置信区间而非单一数值
```

### R-04: Flutter K 线图性能问题

**基本信息**

| 字段 | 内容 |
|------|------|
| 风险 ID | R-04 |
| 风险类别 | 技术风险 |
| 发现日期 | 2026-04-21 |
| 状态 | 🟡 观察中 |
| 负责人 | @frontend |

**风险描述**

K 线图需要渲染大量数据点（每条 K 线包含开高低收 4 个数据 + 成交量），多币种同时监控可能导致性能问题。

**影响分析**

| 维度 | 评估 |
|------|------|
| 影响程度 | 🟡 中 - 影响用户体验 |
| 发生概率 | 🟡 中 - 取决于实现方式 |
| 风险值 | 🟡 6 (中) |

**缓解策略**

```markdown
## 缓解措施

### 1. 渲染优化
- [ ] 使用 Canvas 直接绘制而非 Widget 堆叠
- [ ] 实现数据分页加载
- [ ] 使用 RepaintBoundary 隔离重绘区域

### 2. 数据优化
- [ ] 减少实时更新的数据点
- [ ] 使用压缩算法减少传输数据量
- [ ] 本地缓存 K 线数据

### 3. 降级方案
- [ ] 低性能设备使用简化版图表
- [ ] 限制同时渲染的 K 线数量
```
