| 维度 | 评分 | 说明 |

## 二、需要简化的模块

### 2.1 部署架构（高优先级）

**当前方案**：
```
Kubernetes Cluster + Istio + NATS + TimescaleDB
```

**问题**：
- K8s 学习曲线陡峭，运维复杂
- Istio 过于重量级
- NATS/Kafka 需要专门的运维知识
- TimescaleDB 增加了部署复杂度

**建议简化方案**：

| 阶段 | 推荐方案 | 说明 |
|------|----------|------|
| **MVP** | Docker Compose + 单机部署 | 快速交付，运维简单 |
| **成长期** | Docker Compose + 云服务器 | 自动扩容交给云厂商 |
| **成熟期** | K8s + 微服务 | 确有必要时再升级 |

**具体改动**：

```
# 原方案
├── Kubernetes Cluster
│   ├── API Pod (x3)
│   ├── Strategy Engine (x5)
│   ├── Risk Engine (x3)
│   └── Market Data (x2)
├── NATS / Kafka
└── TimescaleDB

# 建议 MVP 方案
├── 单台云服务器 (2C4G)
│   ├── FastAPI App (所有服务)
│   └── Redis + PostgreSQL
└── 第三方 K 线数据（如 CoinGecko）
```

### 2.2 服务拆分（中优先级）

**当前方案**：10+ 微服务

**建议**：MVP 阶段合并为 3-4 个服务

```
# 原方案
├── user-service
├── account-service
├── market-data-service
├── strategy-engine
├── execution-service
├── risk-engine
├── backtest-engine
├── notification-service
└── admin-service

# 建议 MVP 方案
├── core-api              # 核心 API（用户+账户+策略）
├── strategy-worker      # 策略执行 Worker
├── market-collector     # 行情采集（可选，后期可改为第三方）
└── admin-api            # 管理后台（可选，后期可与 core 合并）
```

### 2.3 数据库方案（低优先级，可保持）

**当前方案**：PostgreSQL + TimescaleDB + Redis

**建议**：MVP 阶段保持，但可以简化

```
# 建议 MVP 方案
├── PostgreSQL          # 主数据库（用户、策略、订单）
├── TimescaleDB        # K 线数据（如果自建行情）
│                      # 或使用第三方（如 Binance API 直连）
└── Redis              # 缓存 + Session + 限流
```

## 三、关键技术决策建议

### 3.1 是否需要自建行情服务？

**原方案**：自建 WebSocket 聚合服务

**建议**：MVP 使用 Binance 原生 API

```
理由：
1. Binance WebSocket 足够稳定
2. 减少开发和运维成本
3. 后期可扩展到多交易所

实现方式：
- FastAPI WebSocket 直接订阅 Binance
- 或使用 ccxt 库统一接口
```

### 3.2 Go vs Python？

**原方案**：Go（行情服务）+ Python（核心服务）

**建议**：统一使用 Python

```
理由：
1. 团队技术栈统一
2. Python 量化生态丰富（pandas, numpy, ta-lib）
3. MVP 阶段性能不是瓶颈
4. 减少多语言维护成本

性能优化：
- 使用 asyncio 异步处理
- 关键路径用 Cython
- 后期瓶颈可用 Go 重写
```

### 3.3 是否需要服务网格？

**原方案**：Istio

**建议**：MVP 不需要

```
理由：
1. 10人团队，Istio 学习成本高
2. MVP 流量小，不需要复杂流量管理
3. 监控用 Prometheus + Grafana 足够

替代方案：
- Docker Compose + Nginx 反向代理
- 监控：Prometheus + Grafana
- 日志：结构化日志 + Loki
```

## 六、需要保留的设计亮点

尽管建议简化，以下设计仍然值得保留：

1. **多层风控架构** - 金融系统的安全保障
2. **API Key 加密存储** - 安全性设计
3. **事件驱动设计理念** - 虽然实现可以简化，但概念要保留
4. **策略模板模式** - 便于后续扩展
5. **统一 API 规范** - REST + WebSocket
