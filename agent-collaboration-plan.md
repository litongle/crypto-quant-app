### Phase 2：开发↔QA 循环

**核心规则**：
- 每个任务 **必须** 通过QA验证才能推进下一个
- QA失败 → 带反馈循环回开发，最多3次
- 3次失败 → 升级到orchestrator决策

**每个Task的执行流程**：

```
┌──────────────────────────────────────────────┐
│  Step 1: 开发代理实现任务                       │
│  ├── 读取架构文档 + 任务描述                     │
│  ├── 编写代码 + 单元测试                        │
│  └── 标记任务完成                               │
│                                              │
│  Step 2: QA代理验证                            │
│  ├── 功能验证：是否符合任务描述                   │
│  ├── 代码验证：是否符合架构规范                   │
│  ├── 截图取证：UI类任务必须截图                  │
│  └── 输出：PASS / FAIL + 具体反馈              │
│                                              │
│  Step 3: 决策                                 │
│  ├── PASS → 标记任务验证通过 → 下一任务          │
│  ├── FAIL (尝试<3) → 反馈回开发代理 → Step 1    │
│  └── FAIL (尝试≥3) → 升级orchestrator          │
└──────────────────────────────────────────────┘
```

## 八、风险管控

### 高风险项

| 风险 | 等级 | 缓解策略 |
|------|------|----------|
| 交易所API不稳定 | 🔴 高 | 重试机制 + 降级策略 + 多数据源冗余 |
| 实盘下单涉及真金白银 | 🔴 高 | 沙盒环境优先、模拟盘验证、风控硬限制 |
| 回测与实盘差距 | 🟡 中 | 滑点/手续费模拟、Paper Trading过渡 |
| Flutter性能（K线渲染） | 🟡 中 | Canvas直接绘制、数据分页加载 |
| 用户信任问题 | 🟡 中 | 透明化策略逻辑、风险提示、模拟盘体验 |

### 升级机制

```
问题发现 → 尝试解决（1次）
  ↓ 失败
代理自行重试（1次）
  ↓ 失败
升级到orchestrator
  ↓ orchestrator评估
简单问题 → 分配其他代理协助
复杂问题 → 暂停流水线 → 召集相关代理讨论
阻塞性问题 → 降级方案 → 记录技术债
```
